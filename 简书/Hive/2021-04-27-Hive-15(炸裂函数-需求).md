```
SELECT name, range, CAST(persons * 1.0 / SUM(persons) OVER (PARTITION BY name ) AS DECIMAL(9, 4)) AS ratio
FROM (
 SELECT name, range, SUM(person_num) AS persons
 FROM table
 WHERE .... 
 AND d >= '2020-04-12'
 AND d <= '2020-04-20'
 ......
 AND name IN (
 SELECT name
 FROM table
 WHERE .....
 GROUP BY name
 ORDER BY SUM(order_price) DESC
 LIMIT 20
 )
 GROUP BY name, range
) t1
```
使用窗口来写,执行速度发现和in关键字相比速度并没有快多少(限制三年的数据)
思路是先按照name,range group by 获取sum()度量,然后sum(度量)按照name来分组,获取总的度量,之后dense_rank来获取top_N 接着求取占比...也挺简单的,还有一种情况是两种要求取的度量不一致并且group by 的维度也是不一致的...就没法求了,只能老老实实使用in来求了

#####采坑kylin
```
SELECT level, CAST(persons * 1.0 / SUM(persons) OVER () AS DECIMAL(9, 4)) AS ratio
FROM (
    SELECT hotel_level, SUM(person_num) AS persons
    FROM table
    WHERE d >= '2020-04-12' and d<='2020-04-20'
    ...
        AND level IS NOT NULL
        GROUP BY level
        order by persons desc
) t1
```
说明level是字符串类型,有1,2,3,4,5,执行SQL发现乱序了???非常奇怪,难道是隐式转化了???结果是按照level来排序的
![执行结果](https://upload-images.jianshu.io/upload_images/9049859-4c4e82628adb9ddd.png?imageMogr2/auto-orient/strip%7CimageView2/2/w/1240)

如下,改写一下,使用row_number()来实现排序功能
```
SELECT level, CAST(persons * 1.0 / SUM(persons) OVER () AS DECIMAL(9, 4)) AS ratio,row_number() over(order by persons desc ) as rn
FROM (
    SELECT evel, SUM(person_num) AS persons
    FROM table
    WHERE d >= '2020-04-12' and d<='2020-04-20'
     ......
        GROUP BY level
        order by persons desc
) t1
```
发现问题所在kylin order by嵌套必须加limit.....改写如下
```
SELECT level, CAST(persons * 1.0 / SUM(persons) OVER () AS DECIMAL(9, 4)) AS ratio,row_number() over(order by persons desc ) as rn
FROM (
    SELECT evel, SUM(person_num) AS persons
    FROM table
    WHERE d >= '2020-04-12' and d<='2020-04-20'
     ......
        GROUP BY level
        order by persons desc
        limit 5
) t1
```
