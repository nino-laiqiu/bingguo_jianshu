##2.OpenResty的安装
>OpenResty = Nginx + Lua，是⼀个增强的Nginx，可以编写lua脚本实现⾮常灵活的逻辑

>OpenResty 是⼀个基于 Nginx与 Lua 的⾼性能 Web 平台，其内部集成了⼤量精良的 Lua 库、第三⽅模块以及⼤多数的依赖项。⽤于⽅便地搭建能够处理超⾼并发、扩展性极⾼的动态 Web 应⽤、Web 服务和动态⽹关。
OpenResty通过汇聚各种设计精良的 Nginx模块（主要由 OpenResty 团队⾃主开发），从⽽将Nginx有效地变成⼀个强⼤的通⽤ Web 应⽤平台。这样，Web 开发⼈员和系统⼯程师可以使⽤Lua 脚本语⾔调动 Nginx⽀持的各种 C 以及 Lua 模块，快速构造出⾜以胜任 10K 乃⾄ 1000K 以上单机并发连接的⾼性能 Web 应⽤系统。
OpenResty的⽬标是让你的Web服务直接跑在 Nginx服务内部，充分利⽤ Nginx的⾮阻塞 I/O 模型，不仅仅对 HTTP 客户端请求,甚⾄于对远程后端诸如 MySQL、PostgreSQL、Memcached 以及Redis 等都进⾏⼀致的⾼性能响应

**配置伪装图⽚的静态地址**
```
location /log.gif {
 #伪装成gif⽂件
 default_type image/gif;
 #本身关闭access_log
 access_log off;

 #返回空图⽚
 empty_gif;
}
```


**使⽤lua的log_by_lua_file将参数写⼊到指定的⽇志⽂件中**

```
location /log.gif {
 #伪装成gif⽂件
 default_type image/gif;
 #本身关闭access_log
 access_log off;

 #使⽤lua将nginx的接收的参数写⼊到⽇志⽂件中
 log_by_lua_file 'conf/log.lua';

 #返回空图⽚
 empty_gif;
}
```

**在nginx的conf⽬录下创建⼀个log.lua⽂件**
>vi /usr/local/openresty/nginx/conf/log.lua

编写脚本
```
-- 引⼊lua所有解析json的库
local cjson = require "cjson"
-- 获取请求参数列表
local request_args_tab = ngx.req.get_uri_args()
-- 使⽤lua的io打开⼀个⽂件，如果⽂件不存在，就创建，a为append模式
local file = io.open("/logs/access.log", "a")
-- 定义⼀个json对象
local log_json = {}
-- 将参数的K和V迭代出来，添加到json对象中
for k, v in pairs(request_args_tab) do
 log_json[k] = v
end
-- 将json写⼊到指定的log⽂件，末尾追加换⾏
file:write(cjson.encode(log_json), "\n")
-- 将数据写⼊
file:flush()
```
**创建存放⽇志的⽬录并设置写⼊权限**
```
mkdir /logs
chmod o+w /logs
#或者将/logs⽬录的所属⽤户改成nobody
chown -R nobody:nobody /logs
```

**如果⼀直往⼀个⽂件中写⼊数据，这个⽇志⽂件会过⼤，造成读写效率变低，现在按照⼩时⽣成⽂件**

```
-- 引⼊lua⽤来解析json的库
local cjson = require "cjson"
-- 获取请求参数列表
local request_args_tab = ngx.req.get_uri_args()
-- 获取当前系统时间
local time = os.date("%Y%m%d%H",unixtime)
-- 使⽤lua的io打开⼀个⽂件，如果⽂件不存在，就创建，a为append模式
local path = "/logs/access-" .. time .. ".log"
local file = io.open(path, "a")
-- 定义⼀个json对象
local log_json = {}
-- 将参数的K和V迭代出来，添加到json对象中
for k, v in pairs(request_args_tab) do
 log_json[k] = v
end
-- 将json写⼊到指定的log⽂件，末尾追加换⾏
file:write(cjson.encode(log_json), "\n")
-- 将数据写⼊
file:flush()
```

##4.KafkaChannel的优化
>优化1：配置KafkaChannel的ACK机制-1，保证数据不丢，⽽且创建Kafka的topic时，副本数量为2，即数据就保存2份，数据保存2份已经很靠谱了，我们公司的Kafka集群配置是2个cup，24核，48G内存，并且是SSD的磁盘。并且创建topic分区数量的时候，分区的数量为 =(机器的数量 * cpu的数量 * 每个cpu的核数)

>优化2：设置KafkaChannel的压缩和失败重试次数
http://kafka.apache.org/0110/documentation.html#producerconfigs

```
value.serializer=cn._51doit.kafka.ProtoBuffSerializer / Serializer
acks=-1
buffer.memory = 33554432 * 1.5
compression.type = snappy
retries = 3
batch.size = 16384 * 2
max.request.size = (可选)
request.timeout.ms = (可选)
```
```
#配置ACK应答机制
a1.channels.c1.kafka.producer.acks = -1
#失败重试次数
a1.channels.c1.kafka.producer.retries = 10
#是snappy进⾏压缩
a1.channels.c1.kafka.producer.compression.type = snappy
```
>优化3：数据写⼊到Kafka之前，⾃定义序列化器，将数据先序列化，这样可以减少⽹络IO的压⼒ProtoBuff和Kyro、ProtoStuff，现在我们使⽤ProtoBuff和ProtoStuff（其中的⼀种）对数据进⾏序列化

##5.nginx-kafka

>yum install -y git

>cd /usr/local/src

3.进入到librdkafka，然后进行编译

>cd librdkafka
>yum install -y gcc gcc-c++ pcre-devel zlib-devel
./configure
make && make install

4.安装nginx整合kafka的插件，进入到/usr/local/src，clone nginx整合kafka的源码

> cd /usr/local/src
git clone https://github.com/brg-liuwei/ngx_kafka_module

5.进入到nginx的源码包目录下	（编译nginx，然后将将插件同时编译）

>cd /usr/local/src/nginx-1.14.0
	./configure --add-module=/usr/local/src/ngx_kafka_module/
	make
	make install

6.修改nginx的配置文件，详情请查看当前目录的nginx.conf

7.启动zk和kafka集群(创建topic)

>/bigdata/zookeeper-3.4.9/bin/zkServer.sh start
	/bigdata/kafka_2.11-0.10.2.1/bin/kafka-server-start.sh -daemon /bigdata/kafka_2.11-0.10.2.1/config/server.properties

9.加载so库	
>echo "/usr/local/lib" >> /etc/ld.so.conf
	ldconfig
